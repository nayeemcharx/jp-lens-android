## What this app is

**JP Lens** ‚Äî an Android floating-overlay OCR tool for reading Japanese in other apps (games, manga readers, etc.). User grants screen-capture + draw-over-other-apps permissions, then a circular floating button captures the current screen, runs Japanese OCR, and overlays clickable boxes on top of detected text.

**Fully on-device / offline ‚Äî the app makes no network requests** (LLM mode was removed; there is no longer any Anthropic/API integration and no `INTERNET` permission).

Two capture modes (modes `0` word/morpheme, `1` sentence+LLM, and `3` translate-only are all removed ‚Äî their integer ids are unused):
- **MODE_SENTENCE_DICT (2)** ‚Äî boxes per OCR-line-piece, grouped into sentences by Japanese terminators (`„ÄÇÔºÅÔºü!?Ôºé.`); tap any piece for a **fully-offline** breakdown popup (`showDictAnalysisPopup`). **Which sections the popup shows is controlled by three home-screen toggles** (`PREF_SHOW_READING` / `PREF_SHOW_TRANSLATION` / `PREF_SHOW_DICTIONARY`, all default on, read live from prefs on each tap): Dictionary = the Word-by-word list, Reading = the full-sentence kana reading (`JapaneseTokenizer.fullReadingHiragana`), Translation = **offline FuguMT** translation of the full sentence (`Translator`; when the toggle is on but the model is missing/failed, the section says so instead of hiding). Sections whose toggle is off are skipped entirely (no lookup/translate work done). Word-by-word = Sudachi morphemes glossed against the bundled JMdict (`JmDict.lookupWord`), each row showing the word-level **JLPT** (`[N5]`‚Ä¶) when known; **no standalone Kanji section** and no Notes section. **The word-by-word list has one row per morpheme in the exact order they appear in the sentence (no dedup), and each row's label is the morpheme's _surface_ (the text as shown), NOT Sudachi's `normalizedForm`/`base`** ‚Äî which canonicalizes orthographic variants and would turn a kana word into a rarely-used kanji. The label surface lives in `WordEntry.surface`; `WordEntry.word` remains the **lookup form** that keys the gloss, the expand-panel detail, and the Anki front. **Lookup form is kana-aware**: an all-kana surface is looked up by its kana form (base if kana, else the surface's hiragana reading) so JMdict's `kana_pref` ranking surfaces the kana-native entry (and its reading-only / rarely-used-kanji writings); a surface with kanji is looked up by the dictionary `base`. The expand panel still shows JMdict's own writings/readings. **Kana lookups prefer the kana-native word** over a kanji homograph that merely shares the reading (the `kana_pref` ordering ‚Äî see schema below). Each word row has a **‚ñ∏ chevron** that lazily expands a full JMdict detail panel below it ‚Äî all senses with part-of-speech, misc/field/dialect tags, notes, cross-refs, loanword origin, alternate writings/readings (each writing/reading carrying its **JLPT badge** when it's the level-listed spelling), **plus a Kanji subsection for the kanji in that word** (KANJIDIC2 meaning + JLPT via `JmDict.lookupKanji`/`kanjiInfoForWord`) ‚Äî via `JmDict.lookupWordDetail` + `OverlayService.renderWordDetail`; tap again to collapse (expand/collapse does **not** re-anchor the popup ‚Äî it grows in place, and a layout listener clamps it back on-screen if the growth would overflow the bottom edge). The "+" Anki card here uses **front = `WordEntry.word` (the lookup form ‚Äî kanji dictionary form for kanji words, kana for kana-native words)** and **back = full JMdict detail + the word's kanji + the source sentence + its translation, rendered as HTML** (`buildAnkiBackHtml`), not the summary. Requires `assets/jmdict.db` to have been built first (see Build / run).
- **MODE_CROP (4)** ‚Äî no boxes at all: tapping the floating button freezes the current frame and shows a fullscreen crop selector (`showCropSelector` ‚Äî the snapshot in an ImageView + a dim layer with a drag-rectangle cutout + hint + cancel ‚úï). Releasing the drag crops the bitmap to the selection, OCRs **only the crop** (`runCropOcr`), and opens the analysis popup **directly** with *all* the Japanese text found in the crop as one string (`assembleCropText` ‚Äî vertical blocks' lines right-to-left, line text rebuilt from elements; blocks with no Japanese are dropped, but digits/Latin inside a Japanese block are kept; kept blocks are joined with a space; **when the crop is predominantly vertical, the blocks themselves are re-sorted right-to-left, ties top-to-bottom** ‚Äî ML Kit's own block order is ~left-to-right and scrambles multi-column tategaki that got split into separate blocks), anchored at the crop rect; a toast reports when nothing Japanese was found. A selection smaller than ~24dp is treated as accidental and the user can retry. While the selector is up, the floating button stays hidden and `isProcessing` stays true until commit/cancel (`cancelCropSelector` also runs from `handleStart`/`onConfigurationChanged`/`onDestroy` so a stale selector can't wedge the button). The popup honors the same section toggles; with a boxless popup open the floating button shows ‚úï and a tap dismisses it (it must be gone before the next snapshot or it would be captured into it).

Package: `com.nayeemcharx.jplens` (renamed from `com.example.jp_lens_android` for Play Store ‚Äî `com.example.*` is rejected; this `applicationId` is permanent once published). App label is `@string/app_name`.

## Build / run

- **DO NOT build.** Never run `./gradlew` / `gradlew.bat` (assemble, compile, test, etc.) ‚Äî the user builds and runs the app themselves. Make the code changes and stop; don't try to verify by compiling.
- Gradle Kotlin DSL. AGP 9.2.1, Kotlin 2.2.10, Compose BOM 2026.02.01, `compileSdk = 36 (minorApiLevel 1)`, `minSdk = 26` (raised from 24 ‚Äî Sudachi loads its dictionary via `java.nio.file`, which is API 26+), JDK target 11.
- Standard build: `./gradlew :app:assembleDebug` (use `gradlew.bat` on Windows host; in WSL the WSL `./gradlew` works).
- Install + run requires a real device or emulator with **screen-capture permission** ‚Äî the projection prompt is system UI; can't be fully exercised by unit tests.
- `local.properties` holds the SDK path; not committed.
- **Sudachi dictionary (all modes)**: `JapaneseTokenizer` (Sudachi) needs `app/src/main/assets/system_full.dic` (always that filename ‚Äî the app loads it by path regardless of edition), **not committed** (too large). Fetch it once with `python3 scripts/build_sudachi_dict.py` (downloads a prebuilt `sudachi-dictionary-<date>-<edition>.zip` from the WorksApplications S3 site and extracts `system_<edition>.dic` ‚Üí `system_full.dic`; stdlib-only; `--date YYYYMMDD` / `--zip path` to override). **For a small APK, build the `small` edition: `python3 scripts/build_sudachi_dict.py --lite` (~123 MB vs ~360 MB full).** Editions smallest‚Üílargest: `small` < `core` < `full` (`--edition core` is the middle ground); small covers everyday vocab but tokenizes rare words less precisely. The app copies the asset to internal storage on first use; **bump `JapaneseTokenizer.DICT_ASSET_VERSION` when a regenerated dict should replace an already-copied one across in-place updates** (it's `2` now ‚Äî bumped when switching full‚Üísmall). Until the script is run, the popup's word-by-word and reading sections are empty. SudachiDict is Apache-2.0 (lexicon under its own EDRDG/UniDic terms); the extracted `.dic` carries the same attribution.
- **Offline dictionary (word-by-word + expand panels)**: the popup's Dictionary section needs `app/src/main/assets/jmdict.db`, **not committed**. Build it once with `python3 scripts/build_jmdict_db.py` (downloads the latest scriptin/jmdict-simplified release ‚Äî full English JMdict + KANJIDIC2 ‚Äî **plus the JLPT word lists from stephenmk/yomitan-jlpt-vocab** (Jonathan Waller's lists with JMdict entry ids, the same data jisho.org uses), and bakes a SQLite file; stdlib-only; `--jlpt-dir` to use a local copy). **For a small APK, build the *lite* db (~11 MB vs ~102 MB full) ‚Äî common + JLPT-listed words only (the learner-relevant ~10%):** `python3 scripts/build_jmdict_db.py --lite` (from source), or `python3 scripts/build_jmdict_db.py --shrink app/src/main/assets/jmdict.db` to derive the lite subset from an already-built full db **offline** (fast, no download ‚Äî overwrites the asset in place). DB schema (unchanged by lite): `entries(key, reading, gloss, common, word_id, jlpt, kana_pref)` ‚Äî one row per headword spelling: `jlpt` is the **word-level JLPT** (`N5`‚Ä¶`N1`, joined by JMdict id), `kana_pref`=1 when the key is kana **and** the word is kana-native (no normal kanji writing, or a `uk` sense) so a kana surface ranks the true kana word above kanji homographs that share the reading ‚Üí `detail(word_id, json BLOB)` (one **gzip-compressed** trimmed JSON per word, all senses/fields + per-writing/reading `"j"` JLPT tags + word-level `"j"`, for the expand panel) + `tags(code, label)` (JMdict tag-code ‚Üí human label) + `kanji(literal, meanings, jlpt)` + `meta(version)`. The app copies the asset to internal storage on first launch; **bump `JmDict.DB_ASSET_VERSION` whenever the schema or shipped content changes** so devices re-copy across in-place updates (it's `4` now ‚Äî bumped for the lite db). Until the script is run, dict mode shows "Dictionary not built". JMdict/KANJIDIC2 are EDRDG-licensed (CC BY-SA 4.0) and the JLPT lists are Waller's (CC BY 4.0); the derived `.db` carries the same attributions.
- **Offline translation (popup Translation section)**: the FuguMT translator needs the `app/src/main/assets/fugumt/` bundle (`encoder.onnx`, `decoder.onnx`, `source_spm.json`, `vocab.json`, `config.json`), **not committed** (~80 MB). Build it once with `python3 scripts/build_fugumt.py` ‚Äî it downloads `staka/fugumt-ja-en`, exports encoder+decoder to ONNX (**no KV-cache**, `task=text2text-generation`), int8-dynamic-quantizes the weights, and dumps the SentencePiece pieces/scores + Marian vocab + special-token ids. **Unlike the stdlib-only dict scripts, this one needs ML libs**: `pip install "optimum[exporters]" onnx onnxruntime transformers sentencepiece huggingface_hub torch`. Flags: `--fp32` (skip quantization, for A/B-ing quality), `--model PATH` (local model dir), `--out DIR`. The app copies the onnx to internal storage on first use and runs it with ONNX Runtime Mobile (`com.microsoft.onnxruntime:onnxruntime-android`, **1.27.0** ‚Äî earlier versions' `libonnxruntime4j_jni.so` is only 4 KB-aligned and fails Play's 16 KB-page requirement; 1.27.0's `.so` are 16 KB-aligned). The `.onnx` are left **DEFLATE-compressed** in the APK (no `noCompress`) ‚Äî they compress ~30%, which matters for Play's 200 MB download limit, and since Translator copies them to internal storage before opening, compression only costs a one-time inflate. **Bump `Translator.MODEL_ASSET_VERSION`** when a regenerated bundle should re-copy across in-place updates. Until the script is run, translations are omitted (the popup hides the Translation line). FuguMT is Satoshi Takahashi's, CC BY-SA 4.0 (derives from Marian/OPUS-MT, Helsinki-NLP); ONNX Runtime is MIT.

## Architecture details worth knowing

- **Single VirtualDisplay rule** (Android 14+): can only call `createVirtualDisplay()` once per `MediaProjection`. On rotation we `resize()` + swap surface, not recreate. See `setupVirtualDisplay()`.
- **Image -> Bitmap** has row-padding; we crop to `image.width` after `copyPixelsFromBuffer`. Don't reintroduce naive ARGB conversion.
- **Reading order**: vertical Japanese (tategaki) blocks have lines sorted right-to-left by centerX; vertical lines have elements sorted top-to-bottom by `box.top`. `isLineVertical` uses element centroid drift first, falls back to aspect ratio.
- **Char‚Üípixel mapping** (`buildLineRanges`/`rectForRange`): line text is rebuilt by concatenating `element.text` (NOT `line.text`, which can contain hidden separators that break the mapping). Each sentence piece's rect is interpolated inside the tightest overlapping element rects, with a line-box fallback only when no element overlaps.
- **Sentence grouping** (`computeSentenceBoxes`): each terminator flushes a sentence. Pieces of one sentence across lines share `sentenceId` + `fullText` so tapping any piece opens the analysis popup for the same full sentence. **Sentences whose text contains no Japanese characters are dropped entirely** (`JapaneseTokenizer.containsJapanese`) ‚Äî OCR'd Latin/digit UI text gets no box; a "No Japanese text found" toast covers the nothing-rendered case.
- **AnkiDroid flow**: the "+" button sits *leading* (left of the word label) in each word row ‚Äî deliberately, so it stays glued near the word even when the popup is wide (an earlier `weight=1f` on the label pushed it to the far right). Tap ‚Üí if Anki not installed, toast; if permission missing, launch `MainActivity` with `EXTRA_REQUEST_ANKI_PERMISSION` to prompt; else add on a background thread and update the button (`‚úì` green = added, gray = duplicate) + toast. Dedup is keyed on the word (front field) only across all decks of the model.
- **Popup layout invariants** (`buildAnalysisPopup(anchor: Rect, title: String)` ‚Äî used by both the sentence-box tap and crop mode): **populate-then-show** ‚Äî `buildAnalysisPopup` does *not* add the window; `showDictAnalysisPopup` runs all lookups (dict/reading/translation) on a background thread first, fills the sections, then calls `AnalysisPopup.show()`, so the popup appears exactly once, fully populated, at its final size/position. **No loading placeholder, no post-load reposition** (that flicker was the user's complaint). A main-thread token (`popupRequestSeq`) supersedes in-flight lookups on a newer tap, `dismissPopup`, or a new capture, so a stale popup can never surface late. **While the popup is up, the transparent sentence boxes are hidden** (`setSentenceBoxesVisible(false)` in `show()`, restored in `dismissPopup`) ‚Äî visibility toggle only, the box windows stay alive. The window **width is fixed** (`min(screen ‚àí 16dp, 400dp)`) ‚Äî don't reintroduce WRAP_CONTENT width, it caused clipped titles and off-screen initial placement. Height wraps but is capped ~65% of screen by the section ScrollView, whose cap subtracts what the top bar + title actually measured. Structure: top bar (centered grip + ‚úï at right) ‚Üí title line (weighted width, max 4 lines ellipsized, copy icon copies the full text) ‚Üí scrollable sections. Placement prefers above the anchor, falls back below, always clamps on-screen. **Dragging is via the top bar or title row ONLY** ‚Äî a container-wide drag listener was tried and reverted (it fought scroll/click gestures and felt worse); after a drag `userMovedPopup` is set, and a `container` layout listener runs `clampToScreen()` on every height change so chevron expand/collapse can never push the popup past the bottom edge.
- **Crop selection** (`showCropSelector`): the frozen snapshot is shown in a fullscreen overlay window explicitly sized to the real metrics at the physical origin (IN_SCREEN + NO_LIMITS, cutout SHORT_EDGES) so view coords line up 1:1 with snapshot pixels; the commit still adds `getLocationOnScreen` offsets defensively. The dim-with-cutout effect is four plain rects around the selection ‚Äî no xfermode/layers.
- **Threading**: capture/tokenize/OCR/translation all run off the main thread. UI updates (`renderXxxBoxes`, popup `text =` writes) post back to `mainHandler`. The `isProcessing` flag debounces taps on the floating button.
- **Cleanup**: `onDestroy` and `clearMorphemeBoxes`/`clearSentenceBoxes` always `runCatching { windowManager.removeView(...) }` because Android can race a view out from under us during teardown.
- **Live mode switching** (`attachDragAndClick` + `showRadialMenu`): hold the floating button (`MENU_HOLD_MS`, ~0.5s) to open a **half-circle hover-select menu** (the 2 modes + Stop). It's a **drag-and-release** gesture, not tap-to-pick: the *same* finger-down that opened the menu keeps steering it (`updateRadialSelection`), and `ACTION_UP` commits the highlighted option. The semicircle opens **away from the nearer screen edge** ‚Äî button on the left half ‚Üí options fan to the right, on the right half ‚Üí fan left (`faceRight = cx < screenW/2`, 160¬∞ spread) so they never run off-screen. Selection is **angular/directional** (nearest fan angle to the finger, so you only point roughly toward an option); releasing inside the center dead-zone (`radialActivation`) cancels. The menu window is passive (`FLAG_NOT_TOUCHABLE`) ‚Äî the floating button owns the gesture; the scrim is purely visual. Haptics on open (`LONG_PRESS`) and on each hover change (`CLOCK_TICK`). The options are the 2 modes (Êñá sentence, ‚úÇ crop) + Stop. Selecting a mode calls `setMode` (swaps `mode`, clears stale boxes, warms `JmDict`/`Translator`, persists `PREF_LAST_MODE`) ‚Äî no new projection consent (the `MediaProjection` is already granted). A pre-menu drag past `touchSlop` cancels the pending hold so dragging the button to reposition it still works. A fresh session (`handleStart`) always starts in sentence mode.

## Conventions

- Logs all use `Log.i/e(TAG, ...)` with `TAG = "JpLens"` (or `"JpLens.<sub>"` for sub-modules). Grep by `JpLens` in `adb logcat`.
- **No network / no `INTERNET` permission.** The app is fully on-device; the manifest does not declare `INTERNET` (only our own manifest ever did ‚Äî no dependency re-adds it; `ACCESS_NETWORK_STATE` from Google's datatransport telemetry lib is harmless with no `INTERNET`). Keep it that way ‚Äî don't reintroduce a network call without revisiting the Play Data Safety declaration and the privacy copy (in-app `AboutScreen` + `docs/privacy-policy.md`).
- Prefs live in `SharedPreferences("jp_lens")` (`OverlayService.PREFS_NAME`): `PREF_ANKI_DECK` (deck name), `PREF_LAST_MODE` (last capture mode), and the popup-section toggles `PREF_SHOW_READING`/`PREF_SHOW_TRANSLATION`/`PREF_SHOW_DICTIONARY`.
- Packaging excludes `META-INF/{CONTRIBUTORS,LICENSE,NOTICE}.md` because some dependency JARs collide on those.
- AnkiDroid integration needs three manifest entries: the `com.ichi2.anki.permission.READ_WRITE_DATABASE` dangerous permission, a `<queries><package android:name="com.ichi2.anki" /></queries>` block (so the package is visible on Android 11+), and the JitPack repo in `settings.gradle.kts` (the only place `com.github.ankidroid:Anki-Android` is published).

## Things that look weird but are intentional

- `JapaneseTokenizer` drops single ASCII tokens and anything with no Japanese chars ‚Äî keeps the word-by-word list from cluttering with stray Latin/digits. POS blacklist is minimal: symbol/whitespace categories (`Ë£úÂä©Ë®òÂè∑`, `Á©∫ÁôΩ`, `Ë®òÂè∑`, `„Åù„ÅÆ‰ªñ`); the "contains a Japanese char" gate does most of the filtering. There is intentionally no grammatical-glue (particle/auxiliary) filter ‚Äî those still get boxes.
- **All overlay windows carry `FLAG_LAYOUT_IN_SCREEN` (alongside `FLAG_LAYOUT_NO_LIMITS`)** so their `Gravity.TOP|START` origin is the *physical* top-left of the display. The capture is sized via `getRealMetrics` (full physical screen), so OCR pixel coords are full-screen; without `IN_SCREEN` a window's origin is the *content* frame (below the status bar), which made boxes appear offset **down by the status-bar height** in normal apps (Chrome) but aligned in fullscreen-immersive games (status bar hidden ‚Üí content frame == full display). Keep this flag on any new overlay window positioned from OCR coordinates.
- Word-by-word row labels are assembled in `buildWordRow` as `surface (reading) ‚Äî meaning  [JLPT]` (em dash `‚Äî`, U+2014). The reading is shown only when the surface carries kanji and differs from the display; JLPT only when known.

## Common tasks

- **Change the Anki card layout**: `OverlayService.buildAnkiBackHtml` (back-side HTML), and in `AnkiDroidHelper` the `MODEL_FIELDS`/`QFMT`/`AFMT` (note type) + `DEFAULT_DECK_NAME`/`MODEL_NAME`.
- **Change overlay box color/style**: `makeBoxDrawable` (morpheme), `makeSentenceBoxDrawable` (sentence), `updateOcrButtonAppearance` (floating button).
- **Change OCR engine**: swap `JapaneseTextRecognizerOptions` in `recognizer` lazy. Box layout assumes ML Kit's `TextBlock`/`Line`/`Element` hierarchy ‚Äî switching to a different OCR shape means rewriting `morphemeBoxesForLine` and `computeSentenceBoxes`.
- **Adjust sentence terminators**: edit the `terminators` set in `computeSentenceBoxes`.
