## What this app is

**JP Lens** ŌĆö an Android floating-overlay OCR tool for reading Japanese in other apps (games, manga readers, etc.). User grants screen-capture + draw-over-other-apps permissions, then a circular floating button captures the current screen, runs Japanese OCR, and overlays clickable boxes on top of detected text.

**Fully on-device / offline ŌĆö the app makes no network requests** (LLM mode was removed; there is no longer any Anthropic/API integration and no `INTERNET` permission).

Two capture modes (modes `0` word/morpheme, `1` sentence+LLM, and `3` translate-only are all removed ŌĆö their integer ids are unused):
- **MODE_SENTENCE_DICT (2)** ŌĆö user-facing name is **"Full screen mode"** (a drawn corner-bracket icon `CornerBracketDrawable` on the floating button and radial menu ŌĆö formerly a ­¤öŹ glyph; internal names still say "sentence"). Boxes per OCR-line-piece, grouped into sentences by Japanese terminators (`ŃĆé’╝ü’╝¤!?’╝Ä.`); tap any piece for a **fully-offline** breakdown popup (`showDictAnalysisPopup`). **The sentence *is* the dictionary**: the popup shows the sentence with every Kuromoji morpheme **individually underlined and tappable**. A custom `SentenceTextView` (inner class) paints a **short, gapped underline under each morpheme** (not one continuous line ŌĆö the gaps are what show the kuromoji splits; multi-line safe via `Layout.getPrimaryHorizontal` per wrapped line), and does its **own geometry hit-testing** (`onTouchEvent` ŌåÆ `wordAt`, reusing the same per-line spans as the underlines so the **last glyph of a wrapped line stays tappable** ŌĆö `LinkMovementMethod`/`ClickableSpan` was removed because it resolved a trailing-edge tap to the next line's offset and missed the char). **Tapping a word expands its JMdict/KANJIDIC2 entry inline** in a rounded detail card just below the sentence, with a **height expand/collapse animation** (`animateExpand`/`animateCollapse`); tapping the same word collapses it, tapping a different word swaps the content. The **selected** word gets a rounded highlight chip + brighter underline (drawn by the view via `selected`) and recolored text (a `ForegroundColorSpan` the setup manages). The card = a header row (`buildWordRow`: leading "+" Anki button + `surface (reading) ŌĆö meaning [JLPT]`) followed by the full detail (`renderWordDetail`): all senses with part-of-speech, misc/field/dialect tags, notes, cross-refs, loanword origin, alternate writings/readings (each carrying its **JLPT badge** when it's the level-listed spelling), **plus a Kanji subsection** (KANJIDIC2 meaning + JLPT via `JmDict.lookupKanji`/`kanjiInfoForWord`). The per-word gloss + detail load **lazily on tap** on a background thread (`JmDict.lookupWord`/`lookupWordDetail`); only the tappable ranges + lookup form are computed up front. **There is no longer a "Word-by-word" section or a Dictionary toggle** ŌĆö the interactive sentence replaced it. **Three home-screen toggles** (`PREF_SHOW_READING` / `PREF_SHOW_ROMAJI` / `PREF_SHOW_TRANSLATION`, all default on, read live per tap) add whole-sentence sections below the sentence: Reading = the full-sentence kana reading (`JapaneseTokenizer.fullReadingHiragana`), Romaji = its Hepburn transliteration (`JapaneseTokenizer.fullRomaji` ŌĆö from each token's IPADIC ńÖ║ķ¤│/pronunciation so particles come out wa/e/o), Translation = **offline FuguMT** translation (`Translator`; when the toggle is on but the model is missing/failed the section says so instead of hiding). (The tapped-sentence panel is called the **"breakdown"** in the UI ŌĆö the word "popup" is avoided user-facing, though internal code/docs still say popup: `showDictAnalysisPopup`, `buildAnalysisPopup`, `AnalysisPopup`, `dismissPopup`.) **Lookup form is kana-aware**: an all-kana surface is looked up by its kana form (base if kana, else the surface's hiragana reading) so JMdict's `kana_pref` ranking surfaces the kana-native entry over a kanji homograph that shares the reading; a surface with kanji is looked up by the dictionary `base`. The displayed word text is the morpheme's *surface* (`WordEntry.surface`); `WordEntry.word` is the **lookup form** that keys the gloss/detail/Anki front. The "+" Anki card uses **front = `WordEntry.word`** and **back = full JMdict detail + the word's kanji + the source sentence + its translation, rendered as HTML** (`buildAnkiBackHtml`). Requires `assets/jmdict.db` to have been built first (see Build / run); without it the sentence just isn't interactive.
- **MODE_CROP (4)** ŌĆö no boxes at all: tapping the floating button freezes the current frame and shows a fullscreen crop selector (`showCropSelector` ŌĆö the snapshot in an ImageView + a dim layer with a drag-rectangle cutout + hint + cancel Ō£Ģ). Releasing the drag crops the bitmap to the selection, OCRs **only the crop** (`runCropOcr`), and opens the analysis popup **directly** with *all* the Japanese text found in the crop as one string (`assembleCropText` ŌĆö vertical blocks' lines right-to-left, line text rebuilt from elements; blocks with no Japanese are dropped, but digits/Latin inside a Japanese block are kept; kept blocks are joined with a space; **when the crop is predominantly vertical, the blocks themselves are re-sorted right-to-left, ties top-to-bottom** ŌĆö ML Kit's own block order is ~left-to-right and scrambles multi-column tategaki that got split into separate blocks), anchored at the crop rect; a toast reports when nothing Japanese was found. A selection smaller than ~24dp is treated as accidental and the user can retry. While the selector is up, the floating button stays hidden and `isProcessing` stays true until commit/cancel (`cancelCropSelector` also runs from `handleStart`/`onConfigurationChanged`/`onDestroy` so a stale selector can't wedge the button). The popup honors the same section toggles; with a boxless popup open the floating button shows Ō£Ģ and a tap dismisses it (it must be gone before the next snapshot or it would be captured into it).

Package: `com.nayeemcharx.jplens` (renamed from `com.example.jp_lens_android` for Play Store ŌĆö `com.example.*` is rejected; this `applicationId` is permanent once published). App label is `@string/app_name`.

## Build / run

- **DO NOT build.** Never run `./gradlew` / `gradlew.bat` (assemble, compile, test, etc.) ŌĆö the user builds and runs the app themselves. Make the code changes and stop; don't try to verify by compiling.
- Gradle Kotlin DSL. AGP 9.2.1, Kotlin 2.2.10, Compose BOM 2026.02.01, `compileSdk = 36 (minorApiLevel 1)`, `minSdk = 26` (raised from 24 ŌĆö Sudachi loads its dictionary via `java.nio.file`, which is API 26+), JDK target 11.
- Standard build: `./gradlew :app:assembleDebug` (use `gradlew.bat` on Windows host; in WSL the WSL `./gradlew` works).
- Install + run requires a real device or emulator with **screen-capture permission** ŌĆö the projection prompt is system UI; can't be fully exercised by unit tests.
- `local.properties` holds the SDK path; not committed.
- **Tokenizer (all modes)**: `JapaneseTokenizer` uses **Kuromoji IPADIC**, whose dictionary is bundled in the JAR ŌĆö so unlike the old Sudachi setup there is **no dictionary asset to build or ship** and the tokenizer always works out of the box. (The old `scripts/build_sudachi_dict.py` + `system_full.dic` asset are dead ŌĆö Kuromoji replaced Sudachi.)
- **jmdict.db is reachability-pruned to IPADIC**: because the app only ever looks a word up by a token IPADIC emits (`Morpheme.base`), any JMdict entry IPADIC can *never* emit as a whole token is dead weight. `scripts/prune_jmdict_unreachable.py` (needs `pip install janome`) drops those ŌĆö a word is kept only if some spelling is an exact IPADIC dictionary entry **or** gets emitted whole in a favorable (punctuation-isolated) sentence context; ~55% of JMdict is unreachable, taking the db from ~98 MB ŌåÆ **~43 MB** with **no visible gloss loss** (unlike the frequency-based `--lite`/`--shrink`, which drops rare-but-reachable words). Run it on the freshly-built full db: `python3 scripts/build_jmdict_db.py` then `python3 scripts/prune_jmdict_unreachable.py`. **Caveat:** the pruned db is valid only for *plain* IPADIC ŌĆö if a Kuromoji user dictionary is ever re-introduced to force new compounds whole, rebuild the full db so those compounds keep their glosses.
- **Offline dictionary (tappable words + detail cards)**: the popup's per-word dictionary needs `app/src/main/assets/jmdict.db`, **not committed**. Build it once with `python3 scripts/build_jmdict_db.py` (downloads the latest scriptin/jmdict-simplified release ŌĆö full English JMdict + KANJIDIC2 ŌĆö **plus the JLPT word lists from stephenmk/yomitan-jlpt-vocab** (Jonathan Waller's lists with JMdict entry ids, the same data jisho.org uses), and bakes a SQLite file; stdlib-only; `--jlpt-dir` to use a local copy). **For a small APK, build the *lite* db (~11 MB vs ~102 MB full) ŌĆö common + JLPT-listed words only (the learner-relevant ~10%):** `python3 scripts/build_jmdict_db.py --lite` (from source), or `python3 scripts/build_jmdict_db.py --shrink app/src/main/assets/jmdict.db` to derive the lite subset from an already-built full db **offline** (fast, no download ŌĆö overwrites the asset in place). DB schema (unchanged by lite): `entries(key, reading, gloss, common, word_id, jlpt, kana_pref)` ŌĆö one row per headword spelling: `jlpt` is the **word-level JLPT** (`N5`ŌĆ”`N1`, joined by JMdict id), `kana_pref`=1 when the key is kana **and** the word is kana-native (no normal kanji writing, or a `uk` sense) so a kana surface ranks the true kana word above kanji homographs that share the reading ŌåÆ `detail(word_id, json BLOB)` (one **gzip-compressed** trimmed JSON per word, all senses/fields + per-writing/reading `"j"` JLPT tags + word-level `"j"`, for the expand panel) + `tags(code, label)` (JMdict tag-code ŌåÆ human label) + `kanji(literal, meanings, jlpt)` + `meta(version)`. The app copies the asset to internal storage on first launch; **bump `JmDict.DB_ASSET_VERSION` whenever the schema or shipped content changes** so devices re-copy across in-place updates (it's `8` now ŌĆö the shipped db is **reachability-pruned** to what IPADIC can emit, ~43 MB; see `scripts/prune_jmdict_unreachable.py`). Until the script is run, the sentence isn't tappable (no per-word dictionary). JMdict/KANJIDIC2 are EDRDG-licensed (CC BY-SA 4.0) and the JLPT lists are Waller's (CC BY 4.0); the derived `.db` carries the same attributions.
- **Offline translation (popup Translation section)**: the FuguMT translator needs the `app/src/main/assets/fugumt/` bundle (`encoder.onnx`, `decoder.onnx`, `source_spm.json`, `vocab.json`, `config.json`), **not committed** (~80 MB). Build it once with `python3 scripts/build_fugumt.py` ŌĆö it downloads `staka/fugumt-ja-en`, exports encoder+decoder to ONNX (**no KV-cache**, `task=text2text-generation`), int8-dynamic-quantizes the weights, and dumps the SentencePiece pieces/scores + Marian vocab + special-token ids. **Unlike the stdlib-only dict scripts, this one needs ML libs**: `pip install "optimum[exporters]" onnx onnxruntime transformers sentencepiece huggingface_hub torch`. Flags: `--fp32` (skip quantization, for A/B-ing quality), `--model PATH` (local model dir), `--out DIR`. The app copies the onnx to internal storage on first use and runs it with ONNX Runtime Mobile (`com.microsoft.onnxruntime:onnxruntime-android`, **1.27.0** ŌĆö earlier versions' `libonnxruntime4j_jni.so` is only 4 KB-aligned and fails Play's 16 KB-page requirement; 1.27.0's `.so` are 16 KB-aligned). The `.onnx` are left **DEFLATE-compressed** in the APK (no `noCompress`) ŌĆö they compress ~30%, which matters for Play's 200 MB download limit, and since Translator copies them to internal storage before opening, compression only costs a one-time inflate. **Bump `Translator.MODEL_ASSET_VERSION`** when a regenerated bundle should re-copy across in-place updates. Until the script is run, translations are omitted (the popup hides the Translation line). FuguMT is Satoshi Takahashi's, CC BY-SA 4.0 (derives from Marian/OPUS-MT, Helsinki-NLP); ONNX Runtime is MIT.

## Architecture details worth knowing

- **Single VirtualDisplay rule** (Android 14+): can only call `createVirtualDisplay()` once per `MediaProjection`. On rotation we `resize()` + swap surface, not recreate. See `setupVirtualDisplay()`.
- **Image -> Bitmap** has row-padding; we crop to `image.width` after `copyPixelsFromBuffer`. Don't reintroduce naive ARGB conversion.
- **Reading order**: vertical Japanese (tategaki) blocks have lines sorted right-to-left by centerX; vertical lines have elements sorted top-to-bottom by `box.top`. `isLineVertical` uses element centroid drift first, falls back to aspect ratio.
- **CharŌåÆpixel mapping** (`buildLineRanges`/`rectForRange`): line text is rebuilt by concatenating `element.text` (NOT `line.text`, which can contain hidden separators that break the mapping). Each sentence piece's rect is interpolated inside the tightest overlapping element rects, with a line-box fallback only when no element overlaps.
- **Sentence grouping** (`computeSentenceBoxes`): each terminator flushes a sentence. Pieces of one sentence across lines share `sentenceId` + `fullText` so tapping any piece opens the analysis popup for the same full sentence. Sentences flow within a **reading unit** (`readingUnits`), not a raw ML Kit block: horizontal blocks map 1:1, but **adjacent vertical blocks are merged into one right-to-left column run** ŌĆö ML Kit usually returns multi-column tategaki as one block *per column* in ~left-to-right order, which used to flush every column as its own "sentence". Merge guard: Ōēź50% vertical overlap + horizontal gap Ōēż ~1.5├Ś median column (line) width, so neighboring bubbles/panels don't fuse; union-find chains a whole run, then all lines in the unit are sorted right-to-left (ties top-to-bottom). **Sentences whose text contains no Japanese characters are dropped entirely** (`JapaneseTokenizer.containsJapanese`) ŌĆö OCR'd Latin/digit UI text gets no box; a "No Japanese text found" toast covers the nothing-rendered case.
- **Box rendering ŌĆö ONE window for all boxes** (`SentenceBoxOverlayView` + `renderSentenceBoxes`): all sentence boxes are drawn by a single fullscreen overlay window (custom `View`, shared `Paint`s in `onDraw`, tap hit-testing in `onTouchEvent`). **Never go back to one window per box** ŌĆö each overlay window costs its own surface + `ViewRootImpl` + a synchronous WMS binder call, which crashed low-RAM devices on text-dense pages (graphics-buffer exhaustion) and made boxes visibly appear/disappear one at a time. Consequences of the single fullscreen window: (1) it consumes *every* touch while visible, so **tapping or swiping empty space dismisses the boxes** (pass-through-to-the-app-below is gone by design ŌĆö the boxes are stale the moment the underlying content moves, and the change-watcher cleared them anyway); (2) it's added after the floating button, so `renderSentenceBoxes` calls `raiseFloatingButton()` (remove+re-add) to keep the button's tap/drag/hold-menu gestures on top; (3) `setSentenceBoxesVisible(false)` (popup open) makes the window INVISIBLE, which also stops it receiving input, so touches outside the popup reach the app below while it's hidden. The window is sized to real metrics at the physical origin like the crop selector, so canvas coords == OCR pixel coords.
- **AnkiDroid flow**: the "+" button sits *leading* (left of the word label) in each word row ŌĆö deliberately, so it stays glued near the word even when the popup is wide (an earlier `weight=1f` on the label pushed it to the far right). Tap ŌåÆ if Anki not installed, toast; if permission missing, launch `MainActivity` with `EXTRA_REQUEST_ANKI_PERMISSION` to prompt; else add on a background thread and update the button (`Ō£ō` green = added, gray = duplicate) + toast. Dedup is keyed on the word (front field) only across all decks of the model.
- **Popup layout invariants** (`buildAnalysisPopup(anchor: Rect, title: String)` ŌĆö used by both the sentence-box tap and crop mode): **populate-then-show** ŌĆö `buildAnalysisPopup` does *not* add the window; `showDictAnalysisPopup` runs all lookups (dict/reading/translation) on a background thread first, fills the sections, then calls `AnalysisPopup.show()`, so the popup appears exactly once, fully populated, at its final size/position. **No loading placeholder, no post-load reposition** (that flicker was the user's complaint). A main-thread token (`popupRequestSeq`) supersedes in-flight lookups on a newer tap, `dismissPopup`, or a new capture, so a stale popup can never surface late. **While the popup is up, the transparent sentence boxes are hidden** (`setSentenceBoxesVisible(false)` in `show()`, restored in `dismissPopup`) ŌĆö visibility toggle only, the box overlay window stays alive (and, being INVISIBLE, receives no input). The window **width is fixed** (`min(screen ŌłÆ 16dp, 400dp)`, floored at 320dp so short sentences still look right) ŌĆö don't reintroduce WRAP_CONTENT width, it caused clipped titles and off-screen initial placement. Height wraps but is capped ~65% of screen by the section ScrollView, whose cap subtracts what the top bar actually measured. Structure: top bar (centered grip + Ō£Ģ at right) ŌåÆ **everything else scrolls together**: the interactive sentence row (tappable morphemes + copy icon), then the tapped-word detail card, then the Reading/Translation sections. The sentence lives *inside* the scroll (not a fixed title line) so a long sentence scrolls instead of being clipped ŌĆö every morpheme stays reachable/tappable. Placement prefers above the anchor, falls back below, always clamps on-screen. **Dragging is via the top bar ONLY** (the sentence row is tappable + scrolls now, so it can't double as a drag handle; a container-wide drag listener was tried and reverted earlier too). After a drag `userMovedPopup` is set, and a `container` layout listener runs `clampToScreen()` on every height change so the word-detail expand/collapse can never push the popup past the bottom edge.
- **Crop selection** (`showCropSelector`): the frozen snapshot is shown in a fullscreen overlay window explicitly sized to the real metrics at the physical origin (IN_SCREEN + NO_LIMITS, cutout SHORT_EDGES) so view coords line up 1:1 with snapshot pixels; the commit still adds `getLocationOnScreen` offsets defensively. The dim-with-cutout effect is four plain rects around the selection ŌĆö no xfermode/layers.
- **Threading**: capture/tokenize/OCR/translation all run off the main thread. UI updates (`renderXxxBoxes`, popup `text =` writes) post back to `mainHandler`. The `isProcessing` flag debounces taps on the floating button.
- **Cleanup**: `onDestroy` and `clearMorphemeBoxes`/`clearSentenceBoxes` always `runCatching { windowManager.removeView(...) }` because Android can race a view out from under us during teardown.
- **Live mode switching** (`attachDragAndClick` + `showRadialMenu`): hold the floating button (`MENU_HOLD_MS`, ~0.5s) to open a **half-circle hover-select menu** (the 2 modes + Stop). It's a **drag-and-release** gesture, not tap-to-pick: the *same* finger-down that opened the menu keeps steering it (`updateRadialSelection`), and `ACTION_UP` commits the highlighted option. The semicircle opens **away from the nearer screen edge** ŌĆö button on the left half ŌåÆ options fan to the right, on the right half ŌåÆ fan left (`faceRight = cx < screenW/2`, 160┬░ spread) so they never run off-screen. Selection is **angular/directional** (nearest fan angle to the finger, so you only point roughly toward an option); releasing inside the center dead-zone (`radialActivation`) cancels. The menu window is passive (`FLAG_NOT_TOUCHABLE`) ŌĆö the floating button owns the gesture; the scrim is purely visual. Haptics on open (`LONG_PRESS`) and on each hover change (`CLOCK_TICK`). The options are the 2 modes (full screen ŌĆö drawn corner-bracket icon `CornerBracketDrawable`, Ō£é crop) + Stop. Selecting a mode calls `setMode` (swaps `mode`, clears stale boxes, warms `JmDict`/`Translator`, persists `PREF_LAST_MODE`) ŌĆö no new projection consent (the `MediaProjection` is already granted). A pre-menu drag past `touchSlop` cancels the pending hold so dragging the button to reposition it still works. A fresh session (`handleStart`) resumes the **last-used mode** (`PREF_LAST_MODE`, full-screen only as the first-run fallback). **The menu window is sized to real metrics at the physical origin with cutout `SHORT_EDGES` (like the crop selector) ŌĆö never revert to `MATCH_PARENT` without a cutout mode**: on punch-hole devices (Redmi/Xiaomi) the system insets such a window below the cutout, shifting every option circle down relative to the button. The fan center comes from the button's `getLocationOnScreen` (its actual position), not its requested `params.x/y`, and all placement clamps use `getRealMetrics` (plain `resources.displayMetrics` can exclude the nav-bar area).

## Conventions

- Logs all use `Log.i/e(TAG, ...)` with `TAG = "JpLens"` (or `"JpLens.<sub>"` for sub-modules). Grep by `JpLens` in `adb logcat`.
- **No network / no `INTERNET` permission.** The app is fully on-device; the manifest does not declare `INTERNET` (only our own manifest ever did ŌĆö no dependency re-adds it; `ACCESS_NETWORK_STATE` from Google's datatransport telemetry lib is harmless with no `INTERNET`). Keep it that way ŌĆö don't reintroduce a network call without revisiting the Play Data Safety declaration and the privacy copy (in-app `AboutScreen` + `docs/privacy-policy.md`).
- Prefs live in `SharedPreferences("jp_lens")` (`OverlayService.PREFS_NAME`): `PREF_ANKI_DECK` (deck name), `PREF_LAST_MODE` (last capture mode), and the whole-sentence breakdown-section toggles `PREF_SHOW_READING`/`PREF_SHOW_ROMAJI`/`PREF_SHOW_TRANSLATION` (the old `PREF_SHOW_DICTIONARY` / Word-by-word toggle is retired ŌĆö the tappable sentence replaced it; the `"show_dictionary"` key is deliberately not reused).
- Packaging excludes `META-INF/{CONTRIBUTORS,LICENSE,NOTICE}.md` because some dependency JARs collide on those.
- AnkiDroid integration needs three manifest entries: the `com.ichi2.anki.permission.READ_WRITE_DATABASE` dangerous permission, a `<queries><package android:name="com.ichi2.anki" /></queries>` block (so the package is visible on Android 11+), and the JitPack repo in `settings.gradle.kts` (the only place `com.github.ankidroid:Anki-Android` is published).

## Things that look weird but are intentional

- `JapaneseTokenizer` drops single ASCII tokens and anything with no Japanese chars ŌĆö keeps the tappable-word set from cluttering with stray Latin/digits. POS blacklist is minimal: symbol/whitespace categories (`ĶŻ£ÕŖ®Ķ©śÕÅĘ`, `ń®║ńÖĮ`, `Ķ©śÕÅĘ`, `ŃüØŃü«õ╗¢`); the "contains a Japanese char" gate does most of the filtering. There is intentionally no grammatical-glue (particle/auxiliary) filter ŌĆö those still get boxes.
- **All overlay windows carry `FLAG_LAYOUT_IN_SCREEN` (alongside `FLAG_LAYOUT_NO_LIMITS`)** so their `Gravity.TOP|START` origin is the *physical* top-left of the display. The capture is sized via `getRealMetrics` (full physical screen), so OCR pixel coords are full-screen; without `IN_SCREEN` a window's origin is the *content* frame (below the status bar), which made boxes appear offset **down by the status-bar height** in normal apps (Chrome) but aligned in fullscreen-immersive games (status bar hidden ŌåÆ content frame == full display). Keep this flag on any new overlay window positioned from OCR coordinates.
- The tapped-word card's header label is assembled in `buildWordRow` as `surface (reading) ŌĆö meaning  [JLPT]` (em dash `ŌĆö`, U+2014). The reading is shown only when the surface carries kanji and differs from the display; JLPT only when known.

## Common tasks

- **Change the Anki card layout**: `OverlayService.buildAnkiBackHtml` (back-side HTML), and in `AnkiDroidHelper` the `MODEL_FIELDS`/`QFMT`/`AFMT` (note type) + `DEFAULT_DECK_NAME`/`MODEL_NAME`.
- **Change overlay box color/style**: the `Paint`s in `SentenceBoxOverlayView` (sentence boxes), `updateOcrButtonAppearance` (floating button).
- **Change OCR engine**: swap `JapaneseTextRecognizerOptions` in `recognizer` lazy. Box layout assumes ML Kit's `TextBlock`/`Line`/`Element` hierarchy ŌĆö switching to a different OCR shape means rewriting `morphemeBoxesForLine` and `computeSentenceBoxes`.
- **Adjust sentence terminators**: edit the `terminators` set in `computeSentenceBoxes`.
