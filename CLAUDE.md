## What this app is

**JP Lens** ŌĆö an Android floating-overlay OCR tool for reading Japanese in other apps (games, manga readers, etc.). User grants screen-capture + draw-over-other-apps permissions, then a circular floating button captures the current screen, runs Japanese OCR, and overlays clickable boxes on top of detected text.

Three capture modes:
- **MODE_MORPHEME (0)** ŌĆö boxes per Sudachi morpheme (SplitMode.C, biggest-meaning units); **tap a single box** ŌåÆ `showWordDetailPopup`, a scrollable popup with the full offline **JMdict** entry for that word (dictionary form + reading title, then all senses/tags/forms + a Kanji subsection ŌĆö same `renderWordDetail` panel dict mode expands inline; requires `assets/jmdict.db`). **Long-press two boxes** to define a range ŌåÆ `showRangePopup` (joined text, full furigana, offline **FuguMT** translation of the span). JMdict + FuguMT are warmed on entering this mode (`handleStart`/`setMode`), like dict mode.
- **MODE_SENTENCE_LLM (1)** ŌĆö boxes per OCR-line-piece, grouped into sentences by Japanese terminators (`ŃĆé’╝ü’╝¤!?’╝Ä.`). Tap any piece to send the full sentence to the Anthropic API (Claude Haiku 4.5) for a structured tutor-style breakdown (Word-by-word / Kanji / Translation / Notes). Each word row has a "+" button that adds it to AnkiDroid as a flashcard.
- **MODE_SENTENCE_DICT (2)** ŌĆö same sentence boxes/popup as LLM mode but **fully offline, no LLM**. Word-by-word = Sudachi morphemes glossed against the bundled JMdict (`JmDict.lookupWord`), each row showing the word-level **JLPT** (`[N5]`ŌĆ”) when known; Translation = **offline FuguMT** translation of the full sentence (`Translator`); **no standalone Kanji section** (unlike LLM mode) and no Notes section. **The word-by-word list has one row per morpheme in the exact order they appear in the sentence (no dedup), and each row's label is the morpheme's _surface_ (the text as shown), NOT Sudachi's `normalizedForm`/`base`** ŌĆö which canonicalizes orthographic variants and would turn a kana word into a rarely-used kanji. The label surface lives in `WordEntry.surface`; `WordEntry.word` remains the **lookup form** that keys the gloss, the expand-panel detail, and the Anki front. **Lookup form is kana-aware**: an all-kana surface is looked up by its kana form (base if kana, else the surface's hiragana reading) so JMdict's `kana_pref` ranking surfaces the kana-native entry (and its reading-only / rarely-used-kanji writings); a surface with kanji is looked up by the dictionary `base`. The expand panel still shows JMdict's own writings/readings. **Kana lookups prefer the kana-native word** over a kanji homograph that merely shares the reading (the `kana_pref` ordering ŌĆö see schema below). Each word row has a **Ō¢Ė chevron** that lazily expands a full JMdict detail panel below it ŌĆö all senses with part-of-speech, misc/field/dialect tags, notes, cross-refs, loanword origin, alternate writings/readings (each writing/reading carrying its **JLPT badge** when it's the level-listed spelling), **plus a Kanji subsection for the kanji in that word** (KANJIDIC2 meaning + JLPT via `JmDict.lookupKanji`/`kanjiInfoForWord`) ŌĆö via `JmDict.lookupWordDetail` + `OverlayService.renderWordDetail`; tap again to collapse (expand/collapse does **not** reposition the popup ŌĆö it stays anchored top-left and grows). The "+" Anki card here uses **front = `WordEntry.word` (the lookup form ŌĆö kanji dictionary form for kanji words, kana for kana-native words)** and **back = full JMdict detail + the word's kanji + the source sentence + its translation, rendered as HTML** (`buildAnkiBackHtml`), not the summary. Requires `assets/jmdict.db` to have been built first (see Build / run).

Package: `com.nayeemcharx.jplens` (renamed from `com.example.jp_lens_android` for Play Store ŌĆö `com.example.*` is rejected; this `applicationId` is permanent once published). App label is `@string/app_name`.

## Build / run

- **DO NOT build.** Never run `./gradlew` / `gradlew.bat` (assemble, compile, test, etc.) ŌĆö the user builds and runs the app themselves. Make the code changes and stop; don't try to verify by compiling.
- Gradle Kotlin DSL. AGP 9.2.1, Kotlin 2.2.10, Compose BOM 2026.02.01, `compileSdk = 36 (minorApiLevel 1)`, `minSdk = 26` (raised from 24 ŌĆö Sudachi loads its dictionary via `java.nio.file`, which is API 26+), JDK target 11.
- Standard build: `./gradlew :app:assembleDebug` (use `gradlew.bat` on Windows host; in WSL the WSL `./gradlew` works).
- Install + run requires a real device or emulator with **screen-capture permission** ŌĆö the projection prompt is system UI; can't be fully exercised by unit tests.
- `local.properties` holds the SDK path; not committed.
- **Sudachi dictionary (all modes)**: `JapaneseTokenizer` (Sudachi) needs `app/src/main/assets/system_full.dic` (always that filename ŌĆö the app loads it by path regardless of edition), **not committed** (too large). Fetch it once with `python3 scripts/build_sudachi_dict.py` (downloads a prebuilt `sudachi-dictionary-<date>-<edition>.zip` from the WorksApplications S3 site and extracts `system_<edition>.dic` ŌåÆ `system_full.dic`; stdlib-only; `--date YYYYMMDD` / `--zip path` to override). **For a small APK, build the `small` edition: `python3 scripts/build_sudachi_dict.py --lite` (~123 MB vs ~360 MB full).** Editions smallestŌåÆlargest: `small` < `core` < `full` (`--edition core` is the middle ground); small covers everyday vocab but tokenizes rare words less precisely. The app copies the asset to internal storage on first use; **bump `JapaneseTokenizer.DICT_ASSET_VERSION` when a regenerated dict should replace an already-copied one across in-place updates** (it's `2` now ŌĆö bumped when switching fullŌåÆsmall). Until the script is run, morpheme mode shows no boxes and dict-mode word-by-word is empty. SudachiDict is Apache-2.0 (lexicon under its own EDRDG/UniDic terms); the extracted `.dic` carries the same attribution.
- **Offline dictionary (dict + OCR single-tap)**: `MODE_SENTENCE_DICT` (word-by-word) **and** `MODE_MORPHEME`'s single-tap detail popup need `app/src/main/assets/jmdict.db`, **not committed**. Build it once with `python3 scripts/build_jmdict_db.py` (downloads the latest scriptin/jmdict-simplified release ŌĆö full English JMdict + KANJIDIC2 ŌĆö **plus the JLPT word lists from stephenmk/yomitan-jlpt-vocab** (Jonathan Waller's lists with JMdict entry ids, the same data jisho.org uses), and bakes a SQLite file; stdlib-only; `--jlpt-dir` to use a local copy). **For a small APK, build the *lite* db (~11 MB vs ~102 MB full) ŌĆö common + JLPT-listed words only (the learner-relevant ~10%):** `python3 scripts/build_jmdict_db.py --lite` (from source), or `python3 scripts/build_jmdict_db.py --shrink app/src/main/assets/jmdict.db` to derive the lite subset from an already-built full db **offline** (fast, no download ŌĆö overwrites the asset in place). DB schema (unchanged by lite): `entries(key, reading, gloss, common, word_id, jlpt, kana_pref)` ŌĆö one row per headword spelling: `jlpt` is the **word-level JLPT** (`N5`ŌĆ”`N1`, joined by JMdict id), `kana_pref`=1 when the key is kana **and** the word is kana-native (no normal kanji writing, or a `uk` sense) so a kana surface ranks the true kana word above kanji homographs that share the reading ŌåÆ `detail(word_id, json BLOB)` (one **gzip-compressed** trimmed JSON per word, all senses/fields + per-writing/reading `"j"` JLPT tags + word-level `"j"`, for the expand panel) + `tags(code, label)` (JMdict tag-code ŌåÆ human label) + `kanji(literal, meanings, jlpt)` + `meta(version)`. The app copies the asset to internal storage on first launch; **bump `JmDict.DB_ASSET_VERSION` whenever the schema or shipped content changes** so devices re-copy across in-place updates (it's `4` now ŌĆö bumped for the lite db). Until the script is run, dict mode shows "Dictionary not built". JMdict/KANJIDIC2 are EDRDG-licensed (CC BY-SA 4.0) and the JLPT lists are Waller's (CC BY 4.0); the derived `.db` carries the same attributions.
- **Offline translation (dict full-sentence + morpheme range)**: the FuguMT translator needs the `app/src/main/assets/fugumt/` bundle (`encoder.onnx`, `decoder.onnx`, `source_spm.json`, `vocab.json`, `config.json`), **not committed** (~80 MB). Build it once with `python3 scripts/build_fugumt.py` ŌĆö it downloads `staka/fugumt-ja-en`, exports encoder+decoder to ONNX (**no KV-cache**, `task=text2text-generation`), int8-dynamic-quantizes the weights, and dumps the SentencePiece pieces/scores + Marian vocab + special-token ids. **Unlike the stdlib-only dict scripts, this one needs ML libs**: `pip install "optimum[exporters]" onnx onnxruntime transformers sentencepiece huggingface_hub torch`. Flags: `--fp32` (skip quantization, for A/B-ing quality), `--model PATH` (local model dir), `--out DIR`. The app copies the onnx to internal storage on first use and runs it with ONNX Runtime Mobile (`com.microsoft.onnxruntime:onnxruntime-android`, **1.27.0** ŌĆö earlier versions' `libonnxruntime4j_jni.so` is only 4 KB-aligned and fails Play's 16 KB-page requirement; 1.27.0's `.so` are 16 KB-aligned). The `.onnx` are left **DEFLATE-compressed** in the APK (no `noCompress`) ŌĆö they compress ~30%, which matters for Play's 200 MB download limit, and since Translator copies them to internal storage before opening, compression only costs a one-time inflate. **Bump `Translator.MODEL_ASSET_VERSION`** when a regenerated bundle should re-copy across in-place updates. Until the script is run, translations are omitted (the popup hides the Translation line). FuguMT is Satoshi Takahashi's, CC BY-SA 4.0 (derives from Marian/OPUS-MT, Helsinki-NLP); ONNX Runtime is MIT.

## Architecture details worth knowing

- **Single VirtualDisplay rule** (Android 14+): can only call `createVirtualDisplay()` once per `MediaProjection`. On rotation we `resize()` + swap surface, not recreate. See `setupVirtualDisplay()`.
- **Image -> Bitmap** has row-padding; we crop to `image.width` after `copyPixelsFromBuffer`. Don't reintroduce naive ARGB conversion.
- **Reading order**: vertical Japanese (tategaki) blocks have lines sorted right-to-left by centerX; vertical lines have elements sorted top-to-bottom by `box.top`. `isLineVertical` uses element centroid drift first, falls back to aspect ratio.
- **Morpheme positioning**: line text is rebuilt by concatenating `element.text` (NOT `line.text`, which can contain hidden separators that break the charŌåÆpixel mapping). Each morpheme's box is interpolated inside the tightest overlapping element rect, with a line-box fallback only when no element overlaps.
- **Sentence grouping** (`computeSentenceBoxes`): each terminator flushes a sentence. Pieces of one sentence across lines share `sentenceId` + `fullText` so tapping any piece sends the same full sentence to the LLM.
- **AnkiDroid flow**: the "+" button sits *leading* (left of the word label) in each word row ŌĆö deliberately, so it stays glued near the word even when the popup is wide (an earlier `weight=1f` on the label pushed it to the far right). Tap ŌåÆ if Anki not installed, toast; if permission missing, launch `MainActivity` with `EXTRA_REQUEST_ANKI_PERMISSION` to prompt; else add on a background thread and update the button (`Ō£ō` green = added, gray = duplicate) + toast. Dedup is keyed on the word (front field) only across all decks of the model.
- **Range selection** (morpheme mode): long-press = anchor; two anchors = range; third long-press = cancel. `buildRangeText` walks lines in reading order, slicing each line's `lineText` between the anchor `charStart`/`charEnd`.
- **Popup positioning**: try above the box first, fall back below if clipped, horizontal clamp to screen margins. LLM popup additionally caps height at ~65% of screen and re-measures + reflows after the response arrives (see `reposition()` closure). The LLM popup's header row is a drag handle ŌĆö once the user has dragged it, `userMovedPopup` flips and auto-reposition is suppressed so the popup doesn't snap back when the response inflates it.
- **Threading**: capture/tokenize/OCR/HTTP all run off the main thread. UI updates (`renderXxxBoxes`, popup `text =` writes) post back to `mainHandler`. The `isProcessing` flag debounces taps on the floating button.
- **Cleanup**: `onDestroy` and `clearMorphemeBoxes`/`clearSentenceBoxes` always `runCatching { windowManager.removeView(...) }` because Android can race a view out from under us during teardown.
- **Live mode switching** (`attachDragAndClick` + `showRadialMenu`): hold the floating button (`MENU_HOLD_MS`, ~0.5s) to open a **half-circle hover-select menu** (the 3 modes + Stop). It's a **drag-and-release** gesture, not tap-to-pick: the *same* finger-down that opened the menu keeps steering it (`updateRadialSelection`), and `ACTION_UP` commits the highlighted option. The semicircle opens **away from the nearer screen edge** ŌĆö button on the left half ŌåÆ options fan to the right, on the right half ŌåÆ fan left (`faceRight = cx < screenW/2`, 160┬░ spread) so they never run off-screen. Selection is **angular/directional** (nearest fan angle to the finger, so you only point roughly toward an option); releasing inside the center dead-zone (`radialActivation`) cancels. The menu window is passive (`FLAG_NOT_TOUCHABLE`) ŌĆö the floating button owns the gesture; the scrim is purely visual. Haptics on open (`LONG_PRESS`) and on each hover change (`CLOCK_TICK`). Selecting a mode calls `setMode` (swaps `mode`, clears stale boxes, warms `JmDict` for dict/OCR modes, persists `PREF_LAST_MODE`) ŌĆö no new projection consent (the `MediaProjection` is already granted). A pre-menu drag past `touchSlop` cancels the pending hold so dragging the button to reposition it still works.

## Conventions

- Logs all use `Log.i/e(TAG, ...)` with `TAG = "JpLens"` (or `"JpLens.<sub>"` for sub-modules). Grep by `JpLens` in `adb logcat`.
- The Anthropic API key is persisted in `SharedPreferences("jp_lens")` under key `anthropic_api_key` (`OverlayService.PREF_ANTHROPIC_KEY`). The user pastes it into the text field on `MainActivity`.
- Packaging excludes `META-INF/{CONTRIBUTORS,LICENSE,NOTICE}.md` because some dependency JARs collide on those.
- AnkiDroid integration needs three manifest entries: the `com.ichi2.anki.permission.READ_WRITE_DATABASE` dangerous permission, a `<queries><package android:name="com.ichi2.anki" /></queries>` block (so the package is visible on Android 11+), and the JitPack repo in `settings.gradle.kts` (the only place `com.github.ankidroid:Anki-Android` is published).

## Things that look weird but are intentional

- `AnthropicClient.MODEL_ID` is `claude-haiku-4-5` (migrated off AWS Bedrock + gpt-oss-120b to the direct Anthropic Messages API). Keep generic phrasing in any user-facing LLM-mode label if the model changes again.
- `JapaneseTokenizer` drops single ASCII tokens and anything with no Japanese chars ŌĆö keeps morpheme boxes from cluttering with stray Latin/digits. POS blacklist is minimal: symbol/whitespace categories (`ĶŻ£ÕŖ®Ķ©śÕÅĘ`, `ń®║ńÖĮ`, `Ķ©śÕÅĘ`, `ŃüØŃü«õ╗¢`); the "contains a Japanese char" gate does most of the filtering. There is intentionally no grammatical-glue (particle/auxiliary) filter ŌĆö those still get boxes.
- `AnthropicClient.parseSections` falls back to `raw` for the word-by-word section if header parsing finds nothing ŌĆö keeps the popup useful when the LLM ignores the section format.
- **All overlay windows carry `FLAG_LAYOUT_IN_SCREEN` (alongside `FLAG_LAYOUT_NO_LIMITS`)** so their `Gravity.TOP|START` origin is the *physical* top-left of the display. The capture is sized via `getRealMetrics` (full physical screen), so OCR pixel coords are full-screen; without `IN_SCREEN` a window's origin is the *content* frame (below the status bar), which made boxes appear offset **down by the status-bar height** in normal apps (Chrome) but aligned in fullscreen-immersive games (status bar hidden ŌåÆ content frame == full display). Keep this flag on any new overlay window positioned from OCR coordinates.
- Word-by-word lines use an em dash `ŌĆö` (U+2014) separator: `word ŌĆö hiragana ŌĆö meaning ŌĆö [JLPT]`. `parseWordEntries` just splits on `ŌĆö` and trims (no regex) ŌĆö deliberately replaced the old regex parser. A bare `-` in the hiragana slot means "kana-only word, no separate reading". The prompt has no "Phrases" section anymore (folded into Notes); "Kanji" replaced it.

## Common tasks

- **Add a new LLM model**: change `AnthropicClient.MODEL_ID`; check whether the model supports the request shape (Haiku rejects the `effort` param; older models differ on `thinking`/`max_tokens`).
- **Change LLM sections / word format**: edit `buildPrompt`, then keep `parseSections`/`parseWordEntries` and the `Analysis`/`WordEntry` data classes in sync, plus the section views in `OverlayService.showLlmAnalysisPopup`.
- **Change the Anki card layout**: `AnkiDroidHelper.buildBack` (back-side content), `MODEL_FIELDS`/`QFMT`/`AFMT` (note type), `DECK_NAME`/`MODEL_NAME`.
- **Change overlay box color/style**: `makeBoxDrawable` (morpheme), `makeSentenceBoxDrawable` (sentence), `updateOcrButtonAppearance` (floating button).
- **Change OCR engine**: swap `JapaneseTextRecognizerOptions` in `recognizer` lazy. Box layout assumes ML Kit's `TextBlock`/`Line`/`Element` hierarchy ŌĆö switching to a different OCR shape means rewriting `morphemeBoxesForLine` and `computeSentenceBoxes`.
- **Adjust sentence terminators**: edit the `terminators` set in `computeSentenceBoxes`.
